### P1: Prompt utilities (`src/utils/prompt.ts`)

Create `src/utils/prompt.ts` using Node.js built-in `readline/promises`. Zero new dependencies.

**Exports:**

```typescript
/**
 * Ask a text question. Returns defaultValue if user presses Enter without typing.
 * In non-interactive environments (no TTY), returns defaultValue immediately.
 */
export async function ask(question: string, defaultValue?: string): Promise<string>

/**
 * Yes/no confirmation. Returns defaultYes if user presses Enter without typing.
 * In non-interactive environments (no TTY), returns defaultYes immediately.
 */
export async function confirm(question: string, defaultYes?: boolean): Promise<boolean>

/**
 * Numbered selection. Shows options as a numbered list, returns selected option string.
 * In non-interactive environments (no TTY), returns options[defaultIdx] immediately.
 */
export async function select(question: string, options: string[], defaultIdx?: number): Promise<string>
```

**TTY detection:** Check `process.stdin.isTTY`. If not a TTY (piped, CI, non-interactive), return default values immediately without prompting. This ensures `ralph init` piped into a script or running in CI doesn't hang waiting for input.

**Display format:**
```
? Project name (my-app): _
? Accept detected settings? (Y/n): _
? Language:
    1. typescript
  ▸ 2. javascript
    3. python
  [2]: _
```

**Re-export** from `src/utils/index.ts`: `export { ask, confirm, select } from './prompt.js';`

**Tests:** At least 2 tests:
- `ask()` returns default value when input is empty
- `confirm()` returns true for "y", false for "n", default for empty

For testing, the functions should accept an optional `input` readable stream parameter (or use dependency injection) so tests can provide canned input without needing a real TTY. One approach: accept an optional `options` object `{ input?: Readable, output?: Writable }` that defaults to `process.stdin`/`process.stdout`.

**Acceptance criteria:**
- Zero new dependencies (uses `node:readline/promises` only)
- Functions are non-blocking in non-TTY environments
- Testable without a real terminal

### P5: Custom YAML rule autofix

**Current behavior:** Custom YAML rules can only report violations, not fix them.

**New behavior:** YAML rules can define an optional `autofix.replace` field for simple pattern replacement.

**YAML format extension:**

```yaml
name: no-console-log
description: Use structured logger
severity: warning
match:
  pattern: 'console\.log\('
autofix:
  replace: 'logger.info('
fix: Replace console.log with structured logger
```

**Implementation:** In `src/commands/lint/rules/custom-rules.ts`:

1. Extend `CustomRuleDefinition` interface:
   ```typescript
   interface CustomRuleDefinition {
     // ... existing fields
     autofix?: {
       replace: string;
     };
   }
   ```

2. When building the `LintRule`, if `def.autofix?.replace` is defined, add an `autofix` method:
   ```typescript
   autofix(context: LintContext): LintFixResult[] {
     const fixes: LintFixResult[] = [];
     for (const file of context.files) {
       const content = readFileSync(file, 'utf-8');
       if (!pattern.test(content)) continue;
       pattern.lastIndex = 0;
       const updated = content.replace(new RegExp(def.match.pattern, 'g'), def.autofix.replace);
       if (updated !== content) {
         writeFileSync(file, updated);
         fixes.push({ file: relative(context.projectRoot, file), description: `Applied autofix: ${def.name}` });
       }
     }
     return fixes;
   }
   ```

**Tests:**
- Custom YAML rule with `autofix.replace` applies replacement when `--fix` is used
- Custom YAML rule without `autofix` field has no autofix method (existing behavior)
- Replacement is applied globally across matching files

**Acceptance criteria:**
- `autofix.replace` in YAML rules enables `ralph lint --fix` for that rule
- Rules without `autofix` are unchanged
- The regex replacement uses the same `match.pattern` from the rule
- Re-run after fix shows zero violations for auto-fixed rules

### P6: GC test file false positive fix

**Bug:** GC orphaned test detection maps `gc.test.ts` → expects `gc.ts` to exist, but the real source is `index.ts` in the same directory.

**Root cause:** The test-to-source mapping doesn't account for the common pattern where `<dirname>.test.ts` tests `<dirname>/index.ts`.

**Fix:** In `src/commands/gc/index.ts`, in the orphaned test file detection logic, add a check: if the test file's base name (without `.test.ts`) matches the parent directory name, check for `index.ts` (or the appropriate extension) in that directory. If `index.ts` exists, the test is NOT orphaned.

The logic should be approximately:
```typescript
const testBase = fileName.replace(/\.(test|spec)\.(ts|tsx|js|jsx|py|go)$/, '');
const ext = fileName.match(/\.(ts|tsx|js|jsx|py|go)$/)?.[0] ?? '.ts';
const dirName = basename(dirname(testFilePath));

// Check direct match first (existing)
const directSource = join(dirname(testFilePath), testBase + ext);
if (existsSync(directSource)) continue;

// Check if test name matches directory name → likely tests index file
if (testBase === dirName) {
  const indexSource = join(dirname(testFilePath), 'index' + ext);
  if (existsSync(indexSource)) continue;
}

// Also check: test is at same level as index file
const indexInSameDir = join(dirname(testFilePath), 'index' + ext);
if (existsSync(indexInSameDir)) continue;
```

**Tests:**
- `gc.test.ts` in `commands/gc/` with `index.ts` present → NOT flagged as orphaned
- `random.test.ts` with no `random.ts` AND no `index.ts` → still flagged as orphaned

**Acceptance criteria:**
- `ralph gc` on ralph-cli itself no longer reports false orphaned test files
- Genuinely orphaned test files are still detected
- Both `*.test.ts` and `*.spec.ts` patterns are handled

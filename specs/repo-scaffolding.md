# Repo Scaffolding

`ralph init` creates an agent-optimized project structure in an existing or new repository, following the proven structure from OpenAI's harness engineering approach.

## Job

When a developer starts a project that AI agents will build and maintain, they need the repository structured so any agent can orient itself immediately — without the developer manually creating docs, architecture files, and operational guides from scratch.

## What It Creates

### Root-Level Files

```
AGENTS.md                    # Table of contents (~60-100 lines), operational commands
ARCHITECTURE.md              # Domain map, package layering, dependency rules
```

**AGENTS.md** is the entry point. A concise map that tells agents:
- How to build, test, lint, and run the project
- Where to find deeper documentation (pointers into docs/)
- Core operating principles (brief, not exhaustive)

AGENTS.md is operational. It does NOT contain project history, changelogs, or detailed explanations. It is a table of contents, not an encyclopedia.

**ARCHITECTURE.md** is the structural map. Describes:
- Domain boundaries and their responsibilities
- Package/module layering
- Dependency direction rules
- Cross-cutting concerns and how they're accessed

Kept at root level (not inside docs/) because it's a primary navigation document alongside AGENTS.md.

### docs/ Structure

```
docs/
├── design-docs/              # Design documents with verification status
│   ├── index.md              # Catalog of all design docs
│   └── core-beliefs.md       # Agent-first operating principles
├── exec-plans/               # Execution plans (versioned, with decision logs)
│   ├── active/               # Plans currently being worked on
│   ├── completed/            # Finished plans (archived for context)
│   └── tech-debt-tracker.md  # Known technical debt, prioritized
├── generated/                # Auto-generated documentation
│   └── .gitkeep              # Placeholder — populated by agents/tools
├── product-specs/            # Requirement specs (one per topic of concern)
│   └── index.md              # Catalog of all specs
├── references/               # LLM-friendly docs for external tools/libraries
│   └── .gitkeep              # Placeholder — populated as needed
├── DESIGN.md                 # High-level design philosophy and patterns
├── PLANS.md                  # Overview of active and upcoming plans
├── QUALITY_SCORE.md          # Quality grades per domain/layer
├── RELIABILITY.md            # Reliability principles and requirements
└── SECURITY.md               # Security principles and requirements
```

### Domain-Level Docs

Instead of a single `principles.md`, principles are split by concern:

- **`design-docs/core-beliefs.md`** — Agent-first operating principles. How agents should approach this codebase.
- **`DESIGN.md`** — Design philosophy, patterns, and conventions used across the project.
- **`RELIABILITY.md`** — Reliability requirements, error handling conventions, observability expectations.
- **`SECURITY.md`** — Security principles, data handling rules, auth patterns.

Additional domain docs can be added per project needs (e.g., `FRONTEND.md`, `API.md`, `DATA.md`). Each is a standalone document that agents discover via AGENTS.md pointers.

### references/ Directory

Stores LLM-friendly documentation for external tools and libraries the project depends on. These are typically `*-llms.txt` files — condensed, agent-readable summaries of external documentation.

Examples:
- `design-system-reference-llms.txt` — component library API reference
- `nixpacks-llms.txt` — deployment tool reference
- `uv-llms.txt` — package manager reference

Purpose: agents working in the repo can reference these directly instead of searching the internet or hallucinating API details. The developer curates which external docs are important enough to bundle.

See the references spec for full details.

### .ralph/ Configuration

```
.ralph/
├── config.yml               # Project settings, rule definitions, runner config
└── rules/                   # Custom architectural rules
```

## Behavior

- `ralph init` — interactive setup. Asks project name, description, tech stack, preferred agent runner. Generates all files.
- `ralph init --defaults` — non-interactive, uses sensible defaults.
- `ralph init` on an existing project — detects what already exists, fills gaps, doesn't overwrite. Reports what it created vs what it skipped.
- All generated files are plain markdown. No agent-specific syntax. No model names. No provider references.
- Generated AGENTS.md includes a comment header: `<!-- Generated by ralph-cli. Edit freely. -->`
- ARCHITECTURE.md is a skeleton with placeholder sections for the developer to fill in.
- Domain docs (DESIGN.md, RELIABILITY.md, SECURITY.md) start with universal defaults that can be extended per-project.
- `core-beliefs.md` starts with agent-first principles (e.g., "repo is the system of record," "validate data at boundaries," "prefer shared utilities over hand-rolled helpers").
- `tech-debt-tracker.md` starts with a template for logging known debt with priority and context.
- `generated/` and `references/` start empty with .gitkeep files.

## Acceptance Criteria

- Running `ralph init` in an empty git repo produces a complete, valid structure that `ralph doctor` reports as healthy
- Running `ralph init` in a repo with an existing AGENTS.md does not overwrite it
- All generated files contain zero references to specific LLM providers, model names, or context window sizes
- Generated AGENTS.md is under 100 lines
- Generated structure matches what `ralph lint`, `ralph grade`, and `ralph doctor` expect to find
- A developer unfamiliar with ralph-cli can read the generated AGENTS.md and understand the project layout within 2 minutes
- An AI agent given only the generated AGENTS.md can navigate to any doc in the structure by following pointers
- `product-specs/index.md` is generated with instructions on the one-topic-per-file convention
- `references/` directory exists and is documented in AGENTS.md as the place to put external tool docs
- `exec-plans/tech-debt-tracker.md` is generated with a usable template

## Out of Scope

- Generating application code or test scaffolds (that's the agent's job)
- CI/CD setup (see integration spec)
- Git hooks (see integration spec)
- Importing existing documentation from external sources
- Populating references/ with actual external docs (developer curates these)

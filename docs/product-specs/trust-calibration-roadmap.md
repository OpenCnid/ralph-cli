# Trust Calibration Roadmap

**Version:** 0.6.0
**Status:** Draft
**Date:** 2026-03-13

---

## Thesis

Ralph v0.1–v0.5 is a **failure catcher**. Tests pass or fail, scores go up or down, validation gates open or close. That works when AI makes obvious mistakes — the "jagged frontier" where AI is clearly bad at some things and good at others.

As AI becomes uniformly capable ("smooth AI"), failures don't disappear — they hide. Tests pass. Specs are met. The code looks correct. But subtle bugs survive because every individual check says "fine" while the composition, the intent, or the approach is wrong.

Ralph needs to evolve from "did this pass?" to **"should you believe it passed?"**

This roadmap adds five features that shift ralph from failure catching to trust calibration. Each feature is independent but they compound: compositional testing catches integration bugs, adversarial generation catches edge cases, calibration tracking measures whether the catching is working, intent verification checks that correctness matches purpose, and divergence detection flags unexplained changes in approach.

---

## Implementation Order

Features are ordered by: value delivered, implementation difficulty, and dependency on prior work.

| Phase | Feature | Spec | Priority | Effort |
|-------|---------|------|----------|--------|
| 1 | Staged Validation | `staged-validation.md` | P0 | Medium |
| 2 | Adversarial Generation | `adversarial-generation.md` | P0 | Large |
| 3 | Calibration Tracking | `calibration-tracking.md` | P1 | Small |
| 4 | Intent Verification | `intent-verification.md` | P2 | Small |
| 5 | Approach Divergence | `approach-divergence.md` | P2 | Medium |

### Phase 1 — Staged Validation

**Why first:** Everything downstream depends on richer validation signals. Adversarial generation needs to distinguish "unit tests pass but integration broke." Calibration tracking needs per-stage pass/fail data. Without stages, validation is a single binary — too coarse for trust calibration.

**Dependency:** None. Extends existing `validation.ts` and `detect.ts`.

### Phase 2 — Adversarial Generation

**Why second:** This is the highest-impact new primitive. After staged validation provides multi-signal feedback, adversarial generation adds a second agent that actively tries to break the first agent's work. This is where ralph stops asking "did the tests pass?" and starts asking "are the tests good enough?"

**Dependency:** Benefits from staged validation (adversarial tests are a new stage) but can function without it.

### Phase 3 — Calibration Tracking

**Why third:** Once you have richer validation (stages) and active adversarial pressure (fuzzing), you need to measure whether they're working. Calibration tracking adds the meta-layer: how often does ralph say "pass" and mean it? Is trust drifting?

**Dependency:** Reads from `results.tsv`, which stages and adversarial enriched with more signals.

### Phase 4 — Intent Verification

**Why fourth:** Lower-effort supplement. Adjusts spec templates and review prompts to cross-reference implementation against stated motivation, not just requirements. Doesn't need the previous phases but gains context from calibration data.

**Dependency:** None (spec template + prompt change).

### Phase 5 — Approach Divergence

**Why last:** Most experimental. Extends `ralph gc` with temporal pattern analysis. Needs iteration history to compare against, which the earlier phases generate.

**Dependency:** Benefits from richer gc history generated by the run loop.

---

## Non-Goals (Roadmap Level)

- **Human attention decay detection.** Ralph's run loop is autonomous — there's no human review step to measure. This is a process/culture concern, not a CLI feature.
- **Real-time agent confidence scoring.** Current agents don't emit confidence signals. Calibration tracking uses outcome data (pass/fail/discard), not agent self-assessment.
- **Replacing tests with AI judgment.** Trust calibration supplements testing, it doesn't replace it. Tests remain the ground truth.

---

## Success Criteria

The roadmap succeeds when:

1. `ralph run` can distinguish unit failures from integration failures and give agents stage-specific feedback
2. A post-pass adversarial step catches bugs that the builder agent's own tests missed
3. Ralph can report "your last 30 iterations had a 97% pass rate — consider increasing scrutiny" and back it with data
4. The combination of these features produces measurably fewer post-merge bugs than v0.5 in a controlled comparison on an active project
